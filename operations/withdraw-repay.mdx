# Withdraw and Repay

Withdraw reduces supply; repay reduces debt. Withdraw requires account ownership. Repay is intentionally permissionless on the target account.

## Withdraw ABI

```rust
withdraw(
    caller: Address,
    account_id: u64,
    withdrawals: Vec<(Address, i128)>
)
```

| Argument | Meaning |
| --- | --- |
| `caller` | Account owner and token recipient. Must authorize. |
| `account_id` | Existing account ID. |
| `withdrawals` | Batch of `(asset, amount)` withdrawal requests. |

## Withdraw-all sentinel

`amount == 0` means withdraw the full post-accrual position for that asset. The controller maps it to `i128::MAX`; the pool treats any amount greater than or equal to the current actual supply as a full withdrawal.

## Withdraw validation

The controller validates:

- Caller authorization and account ownership.
- Reentry guard is not active.
- Asset position exists.
- Amount is non-negative.
- Market price can be resolved.
- Post-batch health factor remains at least `1e18` when debt remains.

Debt-free withdrawals can use permissive oracle mode. Accounts with debt use strict prices before the final health-factor gate.

## Withdraw accounting

1. Controller loads account metadata and the supply map.
2. Controller loads the borrow map only if debt exists.
3. Controller aggregates withdrawal requests.
4. Controller calls `pool.withdraw(caller, amount, position, false, 0, price_wad)`.
5. Pool syncs indexes and reconstructs current supply.
6. Pool clamps full withdrawals and applies dust-lock protection.
7. Pool checks reserves.
8. Pool burns scaled supply and transfers net tokens to caller.
9. Controller updates or removes the supply position.
10. Controller rechecks health factor.
11. Controller removes the account only if both supply and borrow maps are empty.

## Dust-lock protection

For partial withdrawals, the pool checks whether the residual scaled supply would reconstruct to zero asset units. If so, the pool treats the operation as a full withdrawal and returns the full current actual supply rather than leaving unwithdrawable dust.

## Repay ABI

```rust
repay(
    caller: Address,
    account_id: u64,
    payments: Vec<(Address, i128)>
)
```

| Argument | Meaning |
| --- | --- |
| `caller` | Payer. Must authorize. Does not need to own the target account. |
| `account_id` | Target account ID. |
| `payments` | Batch of `(asset, amount)` repayment requests. |

## Repay validation

The controller validates:

- Caller authorization.
- Payment vector is not empty.
- Positive amounts.
- Borrow position exists.
- Token transfer into the pool succeeds and yields a positive balance delta.

Repay uses permissive oracle mode for non-isolated accounts, but isolated accounts use strict prices because repay updates the global isolated debt counter in USD-WAD.

## Repay accounting

1. Controller transfers tokens from payer to the pool.
2. Controller measures the pool's received balance delta.
3. Controller calls `pool.repay(caller, amount_received, position, price_wad)`.
4. Pool syncs indexes and reconstructs current debt.
5. If payment exceeds debt, pool refunds overpayment to `caller`.
6. Pool burns scaled debt and decrements `borrowed_ray`.
7. Controller updates or removes the borrow position.
8. Controller decrements isolated debt when the account is isolated.
9. Controller writes the borrow-side map.

## Account close behavior

Repay does not delete an account even when it clears all debt. Only owner-controlled withdraw or close-position strategy flows can remove account storage after all supply and borrow positions are gone.

## Failure modes

| Error | Common cause |
| --- | --- |
| `PositionNotFound` | Withdraw or repay references a missing position. |
| `AmountMustBePositive` | Repay amount is zero/negative or transfer delta is zero. |
| `InsufficientLiquidity` | Pool reserves cannot cover withdrawal. |
| `InsufficientCollateral` | Withdrawal would make the account unhealthy. |
| `UnsafePriceNotAllowed` | Strict-price operation sees excessive oracle deviation. |
