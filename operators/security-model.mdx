# Security Model

The protocol is pre-audit and designed around explicit trust boundaries. This page summarizes the live security model for integrators and operators.

## Authorization

| Surface | Gate |
| --- | --- |
| User operations | `caller.require_auth()` and account-owner checks where required. |
| Repay | Payer auth only; target account ownership is intentionally not required. |
| Liquidation | `liquidator.require_auth()` and health-factor eligibility. |
| Owner config | `#[only_owner]`. |
| Keeper, revenue, oracle duties | `#[only_role(caller, "...")]`. |
| Pool mutations | Pool owner auth; owner is controller. |

## Reentry guard

The controller uses `FlashLoanOngoing` as a shared single-flight guard:

- Flash-loan callbacks set it while receiver code executes.
- Strategy aggregator calls set it while router code executes.
- Mutating controller entrypoints reject calls while the guard is true.

A panic rolls back the guard write.

## Oracle risk model

The controller treats Reflector as trusted-but-validated:

- Rejects missing and non-positive prices.
- Rejects stale and excessive future timestamps.
- Requires enough TWAP observations.
- Applies two-tier deviation bands.
- Lets risk-decreasing operations continue on the safe anchor.
- Fails closed for risk-increasing operations.

There is no automatic fallback registry or global oracle circuit breaker. Operators must monitor feed health and manually adjust or disable markets when needed.

## Token assumptions

The accounting model requires listed tokens to behave like fixed-balance 1:1 transfer assets.

Do not list:

- Fee-on-transfer tokens.
- Rebasing tokens.
- Tokens whose decimals can change without coordinated migration.
- Tokens with transfer hooks that can violate expected Soroban auth or balance semantics.

Input-side flows such as supply, repay, and rewards measure received balance deltas. Outgoing flows such as borrow, withdraw, and liquidation rely on standard transfer semantics.

## Pool trust model

Pools are trusted implementation contracts but intentionally narrow:

- They do not perform account-level solvency decisions.
- They cannot mutate controller storage.
- They never call other pools.
- They execute only asset-local accounting.
- They accept mutating calls only from the controller owner path.

## Upgrade risk

Controller and pool upgrades are privileged operations.

Risk areas:

- Storage layout changes have no generic migration framework.
- Existing pools keep their current WASM until upgraded.
- Pool template changes affect future pools, not existing ones.
- Market oracle fields are mandatory for deserialization; breaking changes require coordinated reconfiguration.

Upgrade procedure should include:

1. Pause.
2. Deploy or upload new WASM.
3. Upgrade target contract.
4. Re-run required configuration or migration steps.
5. Read affected market/account/pool views.
6. Run smoke tests.
7. Unpause.

## Strategy router risk

The aggregator is operator-set and external. The controller defends against key router failures:

- Validates swap path shape.
- Forces router sender to the controller.
- Verifies exact input spend.
- Verifies output balance delta.
- Enforces `total_min_out`.
- Blocks reentry during router execution.

The router can still affect execution quality through quotes and route behavior. Operators should monitor router deployments and supported venues.

## Revenue accumulator risk

The accumulator is address-only trust. The controller forwards claimed revenue blindly. If the accumulator is wrong or malicious, revenue can be misrouted. Set and verify it before granting `REVENUE` role or unpausing claims.

## Operational monitoring

Monitor:

- Market status stuck in `PendingOracle`.
- Oracle staleness and deviation errors.
- `protocol_revenue` increasing while `claim_revenue` returns zero.
- Pool reserves below user withdrawal demand.
- Isolated debt near ceiling.
- Supply or borrow caps near saturation.
- Account TTL and shared-state TTL.
- Token issuer upgrades.
- Aggregator contract changes.

## Audit status

The repository documentation states the project is pre-audit, with internal review and remediation complete and external audit by Runtime Verification and Certora as the next milestone. Treat deployments accordingly until external audit status changes.
