# Configuration

Configuration is split between owner-only lifecycle settings and role-gated operational settings.

## Roles

| Role | Granted by | Scope |
| --- | --- | --- |
| Owner | Constructor and two-step transfer | Lifecycle, upgrades, roles, pools, market risk config, e-mode config, token allowlist. |
| `KEEPER` | Constructor grants initial keeper; owner can grant/revoke | Index updates, threshold propagation, bad-debt cleanup, TTL keepalive. |
| `REVENUE` | Owner grants explicitly | Claim revenue and add rewards. |
| `ORACLE` | Owner grants explicitly | Configure or edit market oracle routes and tolerances. |

Only `KEEPER` is granted at construction. `REVENUE` and `ORACLE` require explicit post-deploy grants.

## Global addresses

| Setting | Endpoint | Validation |
| --- | --- | --- |
| Aggregator | `set_aggregator(addr)` | Address must exist and be a WASM contract. |
| Accumulator | `set_accumulator(addr)` | Address must exist and be a WASM contract. |
| Pool template | `set_liquidity_pool_template(hash)` | Hash must be non-zero. |

## Position limits

`set_position_limits(PositionLimits)` controls the max unique supplied assets and borrowed assets per account.

Validation:

- `max_supply_positions > 0`
- `max_borrow_positions > 0`
- each limit must be at most `32`

The upper bound protects liquidation iteration from exhausting transaction budget.

## Token allowlist

| Endpoint | Effect |
| --- | --- |
| `approve_token_wasm(token)` | Allows token address for market creation. |
| `revoke_token_wasm(token)` | Removes approval for future market creation. |

The emitted event uses a hash derived from the token address XDR. The runtime gate is the token address itself.

## Asset config validation

`AssetConfig` is validated before market creation and edits.

Key constraints:

- LTV cannot exceed liquidation threshold.
- Liquidation threshold must be within BPS bounds.
- Liquidation bonus and fees must be within configured bounds.
- Borrow and supply caps must be non-negative.
- Flash-loan fee must be non-negative.
- Isolation debt ceiling must be non-negative.

Operationally, keep LTV below liquidation threshold with enough buffer for oracle movement and interest accrual.

## Interest-rate model validation

Rate updates validate:

- Rates are non-negative.
- `mid_utilization_ray > 0`.
- `optimal_utilization_ray > mid_utilization_ray`.
- `optimal_utilization_ray < RAY`.
- `0 <= reserve_factor_bps < BPS`.
- Slope chain is monotone: `base <= slope1 <= slope2 <= slope3 <= max`.
- `max_borrow_rate_ray` stays inside the compound-interest envelope.

The controller syncs indexes before applying new pool parameters.

## Oracle configuration

`configure_market_oracle(caller, asset, cfg)` validates:

- Market exists.
- Staleness window is within supported bounds.
- Tolerances satisfy first/last ordering and max limits.
- Token decimals can be read on-chain.
- Required CEX feed decimals can be read.
- DEX feed decimals can be read when DEX is configured.
- Tickers can be resolved at configuration time.
- TWAP record count is bounded.

Market activation depends on this step.

## E-mode configuration

| Endpoint | Purpose |
| --- | --- |
| `add_e_mode_category(ltv, threshold, bonus)` | Creates a category with risk parameters. |
| `edit_e_mode_category(id, ltv, threshold, bonus)` | Updates category values. |
| `remove_e_mode_category(id)` | Marks category deprecated and cleans membership indexes. |
| `add_asset_to_e_mode_category(asset, category_id, can_collateral, can_borrow)` | Adds asset membership. |
| `edit_asset_in_e_mode_category(asset, category_id, can_collateral, can_borrow)` | Updates membership flags. |
| `remove_asset_from_e_mode(asset, category_id)` | Removes membership. |

Category validation requires LTV to be non-negative, threshold to exceed LTV, threshold to be at most BPS, and bonus to remain within configured bounds.

## Pause policy

The controller is paused at construction and on upgrade. Operators should unpause only after:

- Aggregator is set, or strategy use is intentionally disabled.
- Accumulator is set before revenue claims.
- Pool template is set.
- Markets are created.
- Oracles are configured.
- E-modes are configured.
- Smoke tests pass.

## Token upgrade monitoring

If a Stellar issuer upgrades an issued-asset SAC and changes decimals or transfer semantics, market accounting assumptions can break. Runbook:

1. Pause immediately.
2. Compare token decimals and transfer semantics against market assumptions.
3. If benign, document and unpause.
4. If breaking, migrate to a new token contract/market.
