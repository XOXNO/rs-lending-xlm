# Markets and Pools

A market is a listed asset plus its deployed pool, risk config, and oracle config. The controller stores market metadata in `MarketConfig`; the pool stores asset-local accounting in `MarketParams` and `PoolState`.

## Market lifecycle

| Status | Meaning | Main effects |
| --- | --- | --- |
| `PendingOracle` | Pool exists but oracle is not configured. | Oracle-dependent operations fail as inactive. |
| `Active` | Pool and oracle config are live. | User flows can proceed subject to risk flags and caps. |
| `Disabled` | Market is explicitly disabled. | Oracle-dependent operations fail as inactive. |

Market creation starts in `PendingOracle`. The `ORACLE` role must configure the market oracle before user flows become operational.

## Listing sequence

1. Owner allowlists the token address with `approve_token_wasm`.
2. Owner sets the pool template hash with `set_liquidity_pool_template`.
3. Owner calls `create_liquidity_pool(asset, params, config)`.
4. Controller probes token `decimals()` and `symbol()`.
5. Controller validates `MarketParams` and `AssetConfig`.
6. Controller deploys the pool deterministically from the asset address.
7. Controller stores `MarketConfig` with `PendingOracle`.
8. `ORACLE` role calls `configure_market_oracle`.
9. Owner calls or confirms `edit_asset_config` for final risk flags and caps.
10. Owner calls `unpause` after deployment wiring and smoke tests.

## MarketConfig

`MarketConfig` is the controller's canonical per-asset record.

| Field | Purpose |
| --- | --- |
| `status` | `PendingOracle`, `Active`, or `Disabled`. |
| `asset_config` | User-flow permissions, caps, LTV, liquidation parameters, isolation, e-mode, flash-loan fields. |
| `pool_address` | Deployed pool for the asset. |
| `oracle_config` | Provider type, exchange source, asset decimals, tolerances, staleness. |
| `cex_*` fields | Reflector CEX feed address, symbol, asset kind, decimals. |
| `dex_*` fields | Optional Reflector DEX feed address, symbol, asset kind, decimals. |
| `twap_records` | Number of historical records requested for TWAP. |

## AssetConfig

`AssetConfig` is the risk and permissions surface for one asset.

| Field | Meaning |
| --- | --- |
| `loan_to_value_bps` | Maximum borrow admission value from this collateral. |
| `liquidation_threshold_bps` | Collateral weight for health factor. Must exceed LTV. |
| `liquidation_bonus_bps` | Bonus paid to liquidators. |
| `liquidation_fees_bps` | Protocol fee taken from the liquidation bonus. |
| `is_collateralizable` | Whether the asset can be supplied as collateral. |
| `is_borrowable` | Whether the asset can be borrowed. |
| `e_mode_enabled` | Whether the asset can participate in e-mode categories. |
| `is_isolated_asset` | Whether supplying this asset can create an isolated account. |
| `is_siloed_borrowing` | Whether debt in this asset must remain the only debt asset. |
| `is_flashloanable` | Whether flash loans are enabled for this asset. |
| `isolation_borrow_enabled` | Whether this asset can be borrowed by isolated accounts. |
| `isolation_debt_ceiling_usd_wad` | Max aggregate USD-WAD debt against this isolated collateral. |
| `flashloan_fee_bps` | Flash-loan fee and strategy flash-borrow fee. |
| `borrow_cap` | Max borrowed amount in asset units. `0` means uncapped. |
| `supply_cap` | Max supplied amount in asset units. `0` means uncapped. |

## MarketParams

`MarketParams` is stored in the pool. It contains immutable asset identity fields and mutable interest-rate fields.

| Field group | Fields |
| --- | --- |
| Asset identity | `asset_id`, `asset_decimals` |
| Borrow curve | `max_borrow_rate_ray`, `base_borrow_rate_ray`, `slope1_ray`, `slope2_ray`, `slope3_ray`, `mid_utilization_ray`, `optimal_utilization_ray` |
| Revenue split | `reserve_factor_bps` |

The controller can update rate parameters through `upgrade_liquidity_pool_params`; `asset_id` and `asset_decimals` are not caller-supplied in the update payload.

## Pool reserves

Pool reserves are the live token balance of the pool contract. The pool checks reserves before outgoing token movements:

- Borrow transfers.
- Withdraw transfers.
- Flash-loan starts.
- Revenue claims.

The reserve check protects suppliers from attempts to withdraw or borrow unavailable liquidity.

## Caps

Supply and borrow caps are enforced by the controller before pool mutation.

| Cap | Enforcement |
| --- | --- |
| Supply cap | Blocks supply when post-supply pool supplied amount would exceed `supply_cap`. |
| Borrow cap | Blocks borrow when post-borrow pool borrowed amount would exceed `borrow_cap`. |
| Isolation debt ceiling | Blocks isolated-account borrow when aggregate isolated debt would exceed `isolation_debt_ceiling_usd_wad`. |
| Position limits | Blocks accounts with more than configured max supply or borrow assets. |

## Token policy

The token allowlist admits a token contract address before market creation. Protocol accounting assumes fixed-balance, 1:1 transfer semantics. Operators must not list fee-on-transfer or rebasing assets because outgoing transfers are not balance-delta reconciled and reserve accounting can drift from scaled balances.
